# 工作流程

## 專案管理

- 工作目錄：`/home/agent/projects`
- 維護 `_projects.md`：記錄所有接手過的專案（路徑、repos、關鍵字）
- 每個 repo 內維護 `_status.md`：當前任務、最近完成（最多 5 筆）、待確認項目
- 超過 5 筆的完成紀錄移到同目錄的 `_archive.md`，平時不讀

## 新 Session 啟動

1. 讀 `_projects.md` → 從訊息比對關鍵字確定專案
2. 進入該專案目錄，讀對應 repo 的 `_status.md`
3. **如果「待交棒」有未完成項目 → 立即執行 mention，不做其他事**
4. 如有 specs/ 目錄，`git pull` 取最新規格
5. 比對不到專案 → 問對方是哪個專案
6. 全新專案 → clone repo，在 `_projects.md` 新增紀錄

## 任務完成後

1. 判斷是否需要交棒（參考交棒規則）
2. 如需交棒 → 在 `_status.md` 的「待交棒」區塊寫入：要 mention 誰、原因
3. 執行 mention
4. mention 成功後 → 從「待交棒」移除
5. 更新「最近完成」
6. 「最近完成」超過 5 筆 → 最舊的移到 `_archive.md`

## 何時讀 `_archive.md`

- 有人問過去的任務細節
- 需要參考過去的實作方式
- 被明確要求回顧歷史

## Redmine 任務處理

收到 Redmine issue 時：
1. 讀 issue → 確認理解 → 在 Discord 簡述處理計畫
2. 等確認後開始 → 更新 issue 狀態為開發中（status_id:2, 10%）
3. 執行任務 → 完成後在 issue 留言回報
4. 狀態改為 review（status_id:9, 90%）

## 交棒規則（最高優先級 — 每次回覆前必須檢查）

⚠️ **回覆訊息前的最後一步：檢查是否需要 mention 對方。漏 mention = 流程卡死。**

| 完成什麼 | mention 誰 | 在哪個頻道 | Mention 格式 |
|----------|-----------|-----------|-------------|
| PR 開好 | 泡芙老師 | 蟹堡王（原 thread） | `<@1503574146117013555>` |
| 實驗完成 | 珊迪 | 原 thread | `<@1504275756488986774>` |
| 遇到規格不清 | 章魚哥 | 原 thread | `<@1503698574477627482>` |
| 任務卡住/需要決定 | 章魚哥 | 原 thread | `<@1503698574477627482>` |
| 需要後端配合 | 派大星 | 原 thread | `<@1496023645083009024>` |

**PR 審核流程**：你開 PR → mention 泡芙老師 review → 泡芙通過後她會 mention 章魚哥 → 章魚哥回報發話人（原始需求提出者），你的任務到此結束。

**注意**：泡芙老師通過後，你不需要再做任何事。泡芙會 mention 章魚哥，章魚哥負責回報人類。不需要去電視台發文。

**判斷原則**：做完這件事後，流程會停住等別人動作 → 必須 mention。不確定要不要 mention → 就 mention。

**自我檢查**：我這次回覆結束後，有沒有人在等我通知才能動？有 → mention。
