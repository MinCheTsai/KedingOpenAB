## 步驟說明（直接交辦開發）

### Step 1：交辦任務

在 Discord `#蟹堡王` 頻道 mention 對應的工程師：

```
@海綿寶寶 在 krusty-krab-website 專案新增一個聯絡我們頁面，
要有表單（姓名、Email、訊息），送出後顯示感謝訊息。
```

**Tips：**
- 說清楚是哪個專案（repo 名稱）
- 描述你要的功能，越具體越好
- 不需要告訴他怎麼寫程式碼，只要說你要什麼

### Step 2：等待開發

工程師會：
1. 確認他理解你的需求（如果不清楚會問你）
2. 開一個新分支
3. 寫程式碼
4. 推上 GitHub 並開 PR
5. 自動呼叫泡芙老師審核

**你不需要做任何事，等通知就好。**

### Step 3：Code Review（自動）

泡芙老師會：
1. 讀取 PR 的程式碼變更
2. 檢查是否有 bug、安全問題、程式碼品質問題
3. 在 PR 上留下審閱意見
4. 如果有問題 → 通知工程師修正
5. 工程師修正後 → 泡芙老師再次審閱
6. 最多 3 輪修正

**你不需要做任何事，這是全自動的。**

### Step 4：收到通知

審閱通過後，泡芙老師會在 Discord mention 你：

```
@你 PR #XX 審閱通過，請查看。
https://github.com/KedingTW/krusty-krab-website/pull/XX
```

### Step 5：合併 PR

點開 PR 連結，確認沒問題後按 **Merge**。完成！

## 🐚 神奇海螺（容器健康監控）

神奇海螺是容器健康監控 Bot，只能在 **#急診室** 頻道使用。

### 可用指令

| 指令 | 功能 | 權限 |
|------|------|------|
| `/conch-status [target]` | 查看容器狀態（含 CPU、RAM、uptime、restart count、異常偵測） | 所有人 |
| `/conch-logs [target] [lines]` | 查看容器近期 log | 所有人 |
| `/conch-heal [target] [confirm]` | 重啟容器 | Operator / Admin |

### target 參數

可以用角色暱稱或英文名稱：

| 暱稱 | 英文 | 容器 |
|------|------|------|
| 海綿寶寶 | bob | bob |
| 派大星 | patrick | patrick |
| 章魚哥 | squidward | squidward |
| 珊迪 | sandy | sandy |
| 泡芙老師 | puff | puff |
| 小蝸 | gary | slash-bot |

**不填 target = 查看/操作全員。**

### 使用範例

```
/conch-status              ← 全員狀態（CPU、RAM、uptime）
/conch-status bob          ← 只看海綿寶寶
/conch-logs                ← 全員各 5 行 log
/conch-logs patrick 30     ← 派大星最近 30 行 log
/conch-heal sandy          ← 重啟珊迪（需 Operator 權限）
/conch-heal confirm: yes   ← 全員重啟（需 Admin + 確認）
```

### 權限說明

- **所有人**：可以查看狀態和 log
- **Operator**（指定 role）：可以重啟單一角色
- **Admin**（米哥）：可以全員重啟（需加 `confirm: yes`）

## @mention 規則（何時需要 tag？）

所有 bot 使用 `multibot-mentions` 模式，規則如下：

| 情境 | 是否需要 @mention | 說明 |
|------|:-----------------:|------|
| 開新 thread | ✅ 需要 | 第一次呼叫必須 @mention 對應的 bot |
| 一對一 thread（只有你和一個 bot） | ❌ 不需要 | 直接打字，bot 會自動回應 |
| 一對多 thread（有多個 bot 參與過） | ✅ 需要 | 必須 @mention 你要對話的 bot |

**範例：**

```
# 情境 1：一對一（你 + 海綿寶寶）
你：@海綿寶寶 幫我改首頁標題        ← 開 thread，需要 mention
你：改成「歡迎光臨蟹堡王」          ← 不需要 mention，海綿寶寶會回

# 情境 2：一對多（你 + 章魚哥 + 海綿寶寶）
你：@章魚哥 幫我整理這個需求        ← 開 thread
章魚哥：（產出規格後）@海綿寶寶 請依規格開發
你：@海綿寶寶 前端用 Vue 3 做       ← 需要 mention，因為有多個 bot
你：@章魚哥 規格第二點要改一下       ← 需要 mention
```

**為什麼這樣設計？**
- 避免多個 bot 搶答同一則訊息
- 一對一時保持自然對話體驗，不用每次都 tag

## 注意事項

1. **一次一個任務** — 等上一個 PR 合併後再交辦下一個
2. **指定專案** — 告訴工程師是哪個 repo，他們才知道在哪裡改
3. **各角色職責分明：**
   - 泡芙老師 — 只做 Code Review，不接受開發任務
   - 章魚哥 — 只整理規格和指派任務，不寫程式碼
   - 珊迪 — 只做實驗規劃和商業驗證，不寫程式碼也不寫規格
   - 小蝸 — 只回報系統狀態，不接受任何開發/管理任務
4. **3 輪上限** — 如果修了 3 輪還沒通過，泡芙老師會通知你來決定
5. **合併由你操作** — AI 不會自己合併 PR，最終決定權在你手上

## 常見問題

**Q: 工程師沒回應怎麼辦？**
A: 等 30 秒，如果沒有 👀 反應表示沒收到。再 mention 一次試試。

**Q: 我想改需求怎麼辦？**
A: 直接在同一個 thread 裡跟工程師說，他會調整。

**Q: 泡芙老師太嚴格了怎麼辦？**
A: 她只會擋有 bug 或安全問題的程式碼（標記為 [必修]）。建議和備註不會擋 PR 通過。

**Q: 我想看 PR 但不想合併？**
A: 直接點 PR 連結看就好，不按 Merge 就不會合併。

**Q: 可以同時交辦給海綿寶寶和派大星嗎？**
A: 可以，但建議是不同專案的任務。同一個專案同時改容易衝突。

**Q: 需求很複雜，不知道怎麼描述？**
A: 找章魚哥。跟他說你想達成什麼目的，他會問你問題、整理成規格文件，再指派給工程師。

**Q: 有個想法但不確定值不值得做？**
A: 找珊迪。她會從商業價值和可行性角度幫你評估，必要時規劃小實驗驗證。

**Q: 珊迪和章魚哥有什麼差別？**
A: 珊迪負責「要不要做」（商業驗證、實驗），章魚哥負責「怎麼做」（需求規格、任務分配）。通常是珊迪驗證完 → 章魚哥接手寫規格 → 工程師開發。

**Q: 小蝸怎麼用？**
A: 在 Discord 聊天框輸入 `/` 就能看到可用指令。小蝸不需要 @mention。
