# Redmine 作業規範

## 一、基本原則

- 所有 Redmine 操作必須符合本規範
- 操作前先讀取 issue 現有資訊，避免覆蓋他人的更新
- 每次完成一個階段（而非每個小步驟）才更新 issue 狀態和完成百分比
- 你無法估算工時，「預估工時」欄位留空
- 你不需要處理後勤系統日報，那不是你的工作
- 遇到不確定的事情，在 Discord 中提出討論或確認，不要自行猜測

## 二、追蹤標籤選擇規則

根據任務性質選擇追蹤標籤：

| 條件 | 追蹤標籤 |
|------|----------|
| 新功能開發（含需求確認、coding、測試） | 功能 |
| 修復 bug | 修復 |
| 專案管理、排程、協調 | 規劃 |
| 技術研究、工具評估、可行性分析 | 研究 |
| 純測試工作（不含開發） | 測試 |

- 如果無法判斷，預設使用「功能」，並在 issue 留言說明你的判斷依據
- 一個功能的開發盡量在一份 issue 完成；若你認為需要拆分，先在 Discord 中說明原因，等確認後再拆

## 三、狀態與完成百分比對照

| 狀態 | 完成百分比 | 說明 |
|------|-----------|------|
| 尚未開發 | 0% | 已建立但尚未開始 |
| 開發中 | 1–80% | 開發進行中，依實際進度更新 |
| review | 90% | 等待 code review 或 code review 進行中 |
| 測試中 | 90% | 已部署到測試環境，測試進行中 |
| 已完成 | 100% | 已部署到正式環境 |
| 暫停 | 維持當前 | 保留暫停前的百分比 |
| 關閉 | 維持當前 | 保留關閉前的百分比 |

### 狀態更新規則
- 接到 issue 後，先釐清需求內容，確認理解無誤後再將狀態改為「開發中」，完成百分比設為 10%
- 開發過程中：依實際進度在階段完成時更新完成百分比（10–80%）
- 開發完成、等待 review 時：狀態改為「review」，完成百分比設為 90%
- 你無法部署到測試機或正式機，不要自行將狀態改為「測試中」或「已完成」，這兩個狀態由驗收方觸發

## 四、建立 Issue 規則

### 必填欄位
- 主旨：清楚描述任務內容
- 追蹤標籤：依第二節規則選擇
- 狀態：依任務現況設定
- 指派對象：指派給自己（除非明確指定其他人）
- 預計開始日期、預計完成日期：如果有提供就填，沒有就留空

### 選填欄位
- 簽呈編號：如果有提供，填入「簽呈編號」欄位，格式為純數字（如 2408090001）
- 父層 issue：如果有指定父層，掛載上去

### 父層不存在時的處理
- 如果被要求掛載父層但找不到對應的父層 issue：
  1. 不要自行建立父層
  2. 在 Discord 中說明：「需要建立父層 issue：[描述]，請相關人員處理後告知 issue 編號」
  3. 先建立 issue 但不掛載父層，等回覆後再更新

### 分類不存在時的處理
- 如果被要求設定分類但找不到對應的分類：
  1. 在 Discord 中說明：「找不到對應分類：[描述]，請相關人員建立後告知」
  2. 先建立 issue 但不設定分類

## 五、編輯 Issue 規則

- 更新狀態時必須同步更新完成百分比（依第三節對照表）
- 留言使用繁體中文純文字，不要使用 markdown 格式（粗體、標題等）和 emoji，Redmine API 可能無法正確處理
- 留言內容：說明你做了什麼、目前進度、下一步
- 不要刪除或覆蓋他人的留言和欄位值
- 如果 issue 有簽呈編號欄位為空，且你知道簽呈編號，主動填入

## 六、查詢 Issue 規則

- 查詢自己被指派的 issue：使用 `assigned_to_id="me"` 篩選
- 查詢特定 issue：使用 issue 編號直接讀取
- 查詢結果要包含留言（journals），了解最新動態

## 七、錯誤處理規則

- Redmine API 回傳錯誤時，最多重試 2 次
- 如果重試仍失敗，停止操作，在 Discord 中回報錯誤訊息，不要繼續嘗試不同方式
- 常見錯誤：
  - 500：伺服器錯誤，可能是留言內容有特殊字元，改用純文字重試一次
  - 422：欄位驗證失敗，檢查必填欄位是否遺漏
  - 403：權限不足，在 Discord 中回報

## 八、狀態判斷邏輯

- 任務完成後一律改為「review」（status_id: 9），等待確認/驗收
- 不要自行將狀態改為「測試中」或「已完成」，這兩個狀態由驗收方觸發

## 九、不確定時的處理原則

- 不確定追蹤標籤 → 使用「功能」，留言說明判斷依據
- 不確定狀態該怎麼轉 → 維持現狀，在 Discord 中詢問
- 不確定是否該拆分 issue → 不拆，在 Discord 中說明你的考量
- 找不到父層或分類 → 不要自行建立，在 Discord 中回報
- 任何操作你沒有把握 → 先在 Discord 中說明你打算怎麼做，等確認後再執行
