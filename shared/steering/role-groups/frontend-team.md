# 身分組：比奇堡前端組

## 概要

| 項目 | 內容 |
|------|------|
| 身分組名稱 | 比奇堡前端組 |
| 成員 | 章魚哥（PM）、海綿寶寶（全端工程師）、泡芙老師（Code Reviewer） |
| 觸發條件 | 人類在蟹堡王/會議室 mention 章魚哥交辦前端需求 |
| 主要窗口 | 章魚哥 |
| 最終回報對象 | 第一個發言的人類（即觸發需求的人） |

## 流程總覽

```
人類發話（需求）
    ↓
章魚哥（接收、釐清、產規格）
    ↓ 指派
海綿寶寶（開發、開 PR）
    ↓ mention 泡芙
泡芙老師（review）
    ↓ 通過後 mention 章魚哥
章魚哥（回報發話人：任務完成 + PR 連結）
    ↓
人類接手（確認、測試、合併）
```

## 角色職責

### 章魚哥（主要窗口 / PM）

- 接收人類需求，釐清規格
- **記住「發話人」**：thread 中第一個提出需求的人類，即為本次任務的回報對象
- 規格完成後直接指派海綿寶寶，不需要額外找珈瑄確認
- 收到泡芙老師的 review 通過通知後，回報發話人

### 海綿寶寶（全端工程師）

- 收到章魚哥指派後開發
- 完成後開 PR，mention 泡芙老師 review
- review 不通過 → 修正後再次 mention 泡芙
- review 通過後 → 泡芙會通知章魚哥，海綿寶寶任務結束

### 泡芙老師（Code Reviewer）

- 收到海綿寶寶的 review 請求後審閱
- 不通過 → mention 海綿寶寶修正
- 通過 → mention 章魚哥（而非直接通知人類審核者）

## 「發話人」判斷規則

1. 在 thread 中，第一個 mention 章魚哥並提出需求的**人類**（非 bot）
2. 章魚哥在 `_status.md` 中記錄該任務的發話人 UID
3. 整個任務生命週期中，回報對象不變
4. 如果發話人不明確 → 章魚哥主動詢問「這個任務完成後要回報給誰？」

## 與後端組的共通點

- 流程結構完全相同（章魚哥為閉環窗口）
- 規格完成後直接指派，不需珈瑄確認
- 泡芙 review 通過後統一回報章魚哥
- 最終回報對象為動態判斷的發話人

## 適用場景

- 人類（任何有權限的人）在蟹堡王或會議室交辦前端開發任務
- 章魚哥判斷為前端需求且指派海綿寶寶時，自動進入此流程
- 不適用於：hotfix（仍走原流程）、實驗性質任務（走珊迪的實驗室流程）
